# Render deployment

This project is prepared for deployment to Render with Docker, Render Postgres, and Render Key Value.

## Included files

- `render.yaml` — Render Blueprint with:
  - web service
  - background worker
  - Render Postgres
  - Render Key Value
- `scripts/render-predeploy.sh` — clears Laravel caches and runs migrations before each web deploy

## Deployment model

- **Web service** runs the Laravel app over `php artisan serve` inside the Docker image.
- **Worker service** runs `php artisan queue:work`.
- **Database** uses Render Postgres via `DATABASE_URL`.
- **Cache / sessions / queues** use Render Key Value via `REDIS_URL`.

## Important environment behavior

- `APP_KEY` is generated by Render for the web service.
- The worker service reuses `APP_KEY` from the web service through the Blueprint reference.
- `DB_CONNECTION` is set to `pgsql`.
- HTTPS is forced automatically in production / Render runtime by `AppServiceProvider`.

## Deploy steps

1. Push the repository with `render.yaml` to GitHub.
2. In Render, create a new **Blueprint** from the repo.
3. Confirm creation of:
   - `fitclub-pulse-web`
   - `fitclub-pulse-worker`
   - `fitclub-pulse-db`
   - `fitclub-pulse-redis`
4. Wait for the initial deploy to finish.

## Post-deploy checks

- Open `/health`
- Open the main page
- Register/login
- Verify coach/admin pages
- Check worker logs for queue connectivity

## Notes

- Render Postgres is the supported managed database in this setup.
- The local Docker compose setup remains MySQL-based for development.
